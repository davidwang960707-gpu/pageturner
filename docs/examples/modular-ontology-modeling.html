一页读懂

这不是一篇“怎么画类图”的文章，而是一本本体建模的施工手册。

论文通过烹饪食谱案例演示一种模块化方法：先定义用例和能力问题，再选择 Ontology Design Patterns，最后把模式组合成可推理、可维护、可复用的本体。

快读判断：如果你在做企业语义层、知识图谱 schema、行业本体或数据治理标准，这篇适合放进团队建模规范的第一课。

PageTurner 对本体论文的处理重点，是把长段方法论转成可执行流程。下面的五步图就是这类论文最应该被可视化的部分。

Recommended Workflow
1. Use Case明确使用场景和边界，避免一上来就追求“完整世界”。
2. Competency Questions写出本体必须回答的问题，作为建模验收标准。
3. Pattern Selection寻找可复用的 Ontology Design Patterns。
4. Module Assembly把模式实例化为模块，再通过关系组合。
5. Test & Iterate用 CQ 和推理检查模型，再回到用例迭代。

能力问题

Competency Questions 是本体质量的早期验收标准。

如果一个本体回答不了它承诺回答的问题，类和属性再多也只是“看起来很丰富”。PageTurner 会把这类问题抽成清单，方便团队评审。

CQ-01某个食谱需要哪些食材？每种食材需要多少数量和单位？
CQ-02某个步骤之前必须完成哪些步骤？是否存在顺序约束？
CQ-03替换某种食材后，哪些步骤、工具或约束需要更新？
CQ-04哪些食谱共享相同的烹饪动作、工具或食材模式？
CQ-05一个食谱是否满足某种饮食限制或营养条件？

案例拆解

“食谱”只是例子，真正可迁移的是建模动作。

读这篇教程时，不要把注意力停在菜谱领域。它展示的是一种通用动作：把领域知识拆成对象、过程、约束、上下文和数量，再分别寻找成熟模式。

食谱领域企业知识图谱里的等价物建模启发
Recipe业务流程 / 服务蓝图 / 产品方案把整体对象拆成可复用模块
CookingStep审批步骤 / 工艺环节 / 任务节点顺序和依赖需要显式建模
Ingredient输入物料 / 数据字段 / 资源同一对象在不同上下文中角色不同
QuantityKPI / 金额 / 规格参数 / 阈值数值和单位要分离，方便校验与换算
团队落地清单

把这篇论文变成团队规范，可以从 6 个检查点开始。

用例是否写成边界声明？写清楚本体服务什么，不服务什么。
能力问题是否可测试？每个 CQ 都应该能映射到查询或推理检查。
是否复用已有模式？优先找 pattern，再决定是否自定义。
模块边界是否稳定？把变化频率不同的概念拆开。
命名是否一致？类、属性、实例命名必须跨模块可读。
是否保留迭代记录？每次改模型都要能追溯到 CQ 或用例变化。
